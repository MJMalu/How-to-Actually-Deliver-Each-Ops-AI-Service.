Service 01
AI-Powered Systems Design
You map how a business operates, identify where human work can be replaced by AI and automation, and build the system that runs it — connecting tools, logic, and workflows into something that works without constant human input.
$3,000–$8,000 2–4 weeks Notion / ClickUp Make / Zapier Claude MCP
What It Actually Is
Designing operations that run themselves

Most businesses run on habit, memory, and manual effort. Someone checks a spreadsheet every morning. Someone sends the same update email every Friday. A Slack message is how the team knows a project moved stages.

AI-Powered Systems Design replaces that friction with structured logic. You identify the recurring inputs and outputs in a business and design a system where those inputs trigger the right actions automatically — using tools, automations, and Claude as the intelligent layer.

The output isn't a recommendations doc. It's a working operational system the team uses from day one.

Before You Sell This
What you need to know — click to track
Process mapping
How to document a workflow from trigger to output. Map any workflow in numbered steps before you automate anything.
Trigger/action logic
Every automation is: something happens → something else runs. Think in 'when X, then Y' terms across any tool.
One project tool deeply
Notion or ClickUp — know one well enough to build databases, automations, and linked views.
Where AI fits vs. where it doesn't
AI handles judgment, synthesis, generation. Automation handles routing, triggers, repetitive tasks.
Data structure basics
Properties, relations, rollups — understand how fields connect so databases don't break at scale.
Handoff thinking
Every system you build will be used by people who didn't build it. Design for the person who knows nothing.
How to Deliver It
The engagement from start to finish
1
Discovery (Days 1–3)
60–90 min call with the founder or ops lead. Map the 5–8 most recurring workflows. Ask: what triggers it, who does what, what tools are touched, what the output is. Record the call. After, paste the transcript into Claude and ask it to extract and map each workflow.
2
Current State Mapping (Days 3–5)
Create a visual map of how the business currently operates. Share with the client before moving forward. Get sign-off on current state before building — clients often correct things here that change what you build entirely.
3
Future State Design (Days 5–8)
Identify which steps automate, which need Claude, which are unnecessary. Design the future state in the same format. Present as a design doc before building anything. Changes here cost one day; changes during the build cost one week.
4
Build (Days 8–18)
Build in priority order — highest-pain workflow first. Test with real data, not dummy data. Document as you go. Build one workflow at a time, confirm it works, then move to the next.
5
Testing + Handoff (Days 18–22)
Run each automation with a real scenario in the client's actual environment. Watch them use it. Record a Loom walkthrough for every automation built. Run a 60-min handoff session with whoever owns the system day-to-day.
Common Mistake
Building before the client confirms the design. You will spend hours building a system for a workflow the client has already changed in their head. Always get explicit approval on current state and future state design before writing a single automation.

Service 02
Prompt Engineering for Operations
You design the prompts a business uses with Claude — built specifically for their workflows, their voice, and their recurring tasks. Not generic AI use. A bespoke system that turns Claude into their ops assistant.
$500–$3,000 1–2 weeks Claude Notion prompt library
What It Actually Is
Not 'how to use AI' — how to use AI for this business

Most businesses that try AI give up because 'it doesn't know our context.' The root cause: no one designed the inputs. The model is only as useful as the prompt it receives.

This service is about designing a prompt library — a set of structured, reusable prompts built for a specific business's most repeated operational tasks. Brief-to-plan, notes-to-SOP, email-to-update.

The business stops using AI as a search engine and starts using it as an operational layer.

Before You Sell This
What you need to know — click to track
How to write a proper system prompt
Role + context + task + format + constraints. A prompt that produces consistent, on-brand output every time.
Claude-specific behaviour
Claude follows instructions precisely and handles long context well. Know when to use role-setting, examples, and constraints.
Ops task categories
Intake, synthesis, documentation, status reporting, planning, audit, communication — each needs a different structure.
How to test a prompt
Run the same prompt with 3–5 different real inputs. If quality changes significantly, the prompt isn't tight enough.
Notion as a prompt library
Build a searchable prompt database with categories, copy-paste format, usage notes. The library is a product, not a folder.
The client's actual language
Good ops prompts use the client's own terminology. Spend time in their docs before writing a single prompt.
How to Deliver It
The engagement step by step
1
Task audit (Days 1–2)
Ask the client: what are the 10 things you or your team do most often that involve writing, summarising, or organising information? Prioritise by frequency × pain.
2
Collect real examples (Days 2–3)
Ask for 3–5 examples of each task type — past outputs, inputs, emails, notes. You write prompts against real material, not hypotheticals.
3
Write and test (Days 3–8)
Write the first draft of each prompt. Test with 3 real inputs. Is the format right? Tone right? Does it handle edge cases? Refine. Add a fill-in-the-blanks structure so users know exactly what to replace.
4
Build the library (Days 8–10)
Notion database: prompt name, category, when to use it, the prompt (copy-paste ready), sample output, notes. The library is the deliverable — it should feel like a product, not a folder of notes.
5
Team walkthrough (Days 10–12)
45-min session with the people who will use the library. Walk through each category. Run one live with real input during the call. Leave a one-page reference card.
Common Mistake
Writing prompts in a vacuum. Without real examples of the task, your prompts won't match what the team actually does. Always collect source material before writing. The best prompt libraries are built from the client's own existing documents and past outputs.
What the Client Receives
Deliverables
📚
Notion Prompt Library
10–20 tested prompts, organised by function, copy-paste format, with usage notes for each.
🧪
Sample outputs per prompt
At least one real example output so the team knows what 'good' looks like and can calibrate their inputs.
📄
Quick-start reference card
One page: what Claude is, how to use a prompt, how to adapt one when the standard version doesn't quite fit.

Service 03
MCP Integration & Setup
You connect Claude directly to a business's existing tools — Notion, ClickUp, Gmail, calendars, CRMs — so Claude can read, create, and update content in those tools in real time. No more copy-paste between AI and software.
$1,500–$5,000 1–3 weeks Claude Desktop MCP Servers Notion / ClickUp API
What It Actually Is
Making Claude a live participant in the business's tools

MCP (Model Context Protocol) lets Claude connect directly to external tools and data sources. Instead of copying information into Claude and pasting output back out, Claude can read from and write to those tools directly — in real time, within a conversation.

In practice: Claude can search a client's Notion database, create a new page from a brief, update a ClickUp task status, pull this week's calendar and summarise what's coming. All in one conversation, without the user touching any other tool.

Most businesses have the tools. They're even using AI. But the two don't talk to each other — and your job is to fix that.

Before You Sell This
Technical prerequisites — click to track
What MCP is and how it works
MCP lets Claude connect to 'servers' — each is a connector for a specific tool. You configure these in Claude Desktop.
Claude Desktop setup
MCP requires Claude Desktop (not claude.ai browser). Know how to install it and configure claude_desktop_config.json.
API keys and authentication
Most MCP servers require an API key from the tool being connected. Know how to generate these and store them safely.
Which MCP servers exist
Know the main ones: Notion, ClickUp, Google Calendar, Gmail, Airtable. Know where to find new ones (mcp.so, GitHub).
How to test a connection
After setup, verify Claude can read and write to the tool. Have a test script ready: list X, create a test item, update it, confirm in the actual tool.
User training requirements
The person using this setup needs to know: connections can drop, Claude needs specific instructions to use the right tool.
How to Deliver It
The engagement
1
Tool audit (Days 1–2)
List every tool the client uses. Identify which have MCP servers available. Prioritise by: which tools hold the most important data × most painful to work around manually. Don't connect everything — connect the 2–4 with biggest immediate impact.
2
Get credentials (Days 2–4)
Guide the client through generating the required API keys and integration tokens. Do not store these yourself. Document which key belongs to which server and what permissions each one has.
3
Install and configure (Days 3–6)
Install Claude Desktop on the client's machine. Set up the MCP config file. Test each connection individually before testing them together. A broken connection that looks like it's working is worse than one that visibly fails.
4
Build use case prompts (Days 6–10)
For each connected tool, write 3–5 prompts using the connection in a real workflow. Test each one until it works reliably with the client's actual data.
5
Handoff and docs (Days 10–14)
Create a 'how your Claude setup works' guide: what's connected, what each connection can do, the ready-made prompts, and what to do when something breaks. Record the walkthrough.
Common Mistake
Connecting tools the client doesn't actually use daily. MCP setup is impressive — don't let that pull you into connecting six tools when two would change their workflow. Start with the highest-pain 2–4, get those working well, and expand later.

Service 04
SOP Architecture
You design and document the operational processes of a business — not just write them down, but structure them so that anyone who joins the team can execute without explanation. AI makes this 5x faster than it used to be.
$2,000–$6,000 1–3 weeks Claude Notion Loom
What It Actually Is
Turning 'how we do things' into a system that survives without you

Most businesses write SOPs no one reads. They're too long, too vague, or written for the person who already knows how to do the thing.

SOP Architecture means designing the structure first: what processes need documenting, in what format, at what level of detail, stored where, owned by whom. Then documenting them in that structure — using AI to do the heavy writing while you focus on the operational judgment calls.

With Claude, a single consultant can document an entire business's operations in two weeks. The bottleneck is no longer writing speed — it's knowing how to extract the right information.

Before You Sell This
What you need to know — click to track
SOP format types
Step-by-step (sequential), decision tree (if/then), checklist (non-sequential), swim-lane (multiple roles). Know when to use which.
Process extraction interviews
People don't describe processes — they tell stories. Know how to interrupt with 'and then what happens?' until you have a clean sequence.
Writing for a new hire
Every SOP assumes the reader knows the tool but not the context. No jargon without explanation. Say which system, where, and what you're looking for.
Where SOPs live and get used
An SOP no one can find is worthless. Structure the library so the right person finds the right doc without searching.
Using Claude for SOP writing
Feed Claude raw notes or a transcript with your SOP format and let it draft. Review for accuracy. Claude drafts; you validate.
Version control and ownership
SOPs go out of date. Every one needs an owner, a review date, and a version number. Build this in from day one.
How to Deliver It
The engagement
1
Process inventory (Days 1–3)
List every process in the business that needs documenting. Prioritise by: what's highest risk if the person who knows it leaves? What's most frequently done wrong? What slows down onboarding most?
2
Process extraction interviews (Days 3–7)
30 mins per process max. Record everything. Question set: What triggers this? Walk me through every step. What tools at each step? Where are decisions made? What goes wrong? What does done look like?
3
Draft with Claude (Days 7–12)
Paste the transcript into Claude with your SOP template. Let it draft. Review for accuracy: correct sequence, correct tool names, nothing missed. Add screenshots or Loom links where a step needs visual reference.
4
Review with the process owner (Days 12–15)
Ask them to follow the SOP literally — do the process using only the doc, not their memory. Where they get stuck is where the SOP is incomplete. Fix every gap before publishing.
5
Build the library and publish (Days 15–18)
Notion database: name, owner, department, last reviewed, status, version. Publish all reviewed SOPs. Run a 30-min session with the team lead on how to use and maintain the library.
Common Mistake
Writing SOPs that describe what happens, not how to do it. 'The project manager reviews the deliverable' is a job description. 'Open the client's Notion project page. Navigate to the Deliverables database. Open the latest submission. Check it against the brief. If approved, change Status to Approved.' That's an SOP.

Service 05 · Flagship
Operational Audit + Systems Build
The full engagement. You go into a business, diagnose what's operationally broken or missing, design the fix, and build it — delivering a working system with SOPs, automations, and documentation. This is where the $5k–$15k contracts come from.
$5,000–$15,000 4–8 weeks All tools Flagship engagement
What It Actually Is
Diagnosis + fix, not just one or the other

Most ops consultants either do the audit (tell the client what's broken) or the build (implement what someone else designed). This engagement does both — and that's what makes it worth more.

You come in as a diagnostician: interviewing the team, mapping operations, identifying gaps. Then you switch into a builder: designing the new system, building it in their tools, and handing it off with everything they need to run it.

With AI, what used to require a team of three over three months can be delivered by one consultant in 4–8 weeks. That's the business case. That's why the pricing holds.

Before You Sell This
Senior engagement — click to track
All of Services 1–4
This engagement draws on every service in this guide. Don't sell this before you can confidently deliver at least three of the previous four.
How to run a full business audit
Interview across functions — ops, delivery, sales, finance — and synthesise into a clear picture of what's working, what's broken, what's missing.
Scope management
A full audit will surface more problems than you can fix. Scope confidently: here's what we're fixing, here's what we're leaving for phase two.
Stakeholder management
At this project size, multiple stakeholders have different priorities. Know how to run weekly check-ins and handle scope creep without losing trust.
Contracting basics
At $5k+, you need a contract. Minimum: scope, deliverables, timeline, payment terms (50% upfront), out of scope items.
Change of scope framing
'That sounds like a phase two, which I'd love to scope separately after we wrap this.' Name new work as new work — don't absorb it.
How to Deliver It
Full 6-phase engagement
1
Discovery (Week 1)
Interview the founder and 2–3 key team members. Ask about every function: how does work come in, move through the business, go out, get stuck? Look for patterns — the same problem appearing in multiple interviews is your priority.
2
Audit Report (Week 1–2)
Written audit: current state summary, 5–8 gaps with severity ratings (critical / high / medium), root cause for each, and recommended fixes. Use Claude to synthesise your discovery notes into the first draft. Present before building anything.
3
System Design (Week 2–3)
Design the new operational infrastructure: which tools, how they connect, what automations run, what gets documented, how the team interacts with Claude. Produce a design doc and visual map. Changes here cost one day; changes during the build cost one week.
4
Build (Weeks 3–6)
Build in priority order: project management system first, then automations, then SOPs, then AI integrations. Check in with the client every 3–4 days. Share progress as you go — don't disappear for two weeks then resurface with a big reveal.
5
Testing + Refinement (Week 6–7)
Run each component with real work from the business. Have the team use the system for real tasks — not a demo. Collect what breaks or confuses and fix it. One week of real use surfaces things two weeks of testing won't.
6
Handoff (Week 7–8)
Full system documentation. Loom walkthroughs of every component. A 90-min handoff session with the team. A 'first 30 days' guide — what to focus on, what to ignore, what to come back to. And a clear statement of what's out of scope for post-handoff questions.
Common Mistake
Skipping the audit report and going straight to the build. You will build the wrong thing. The audit report aligns you and the client on what the problem actually is before you spend weeks solving it. Every time this step is skipped, the engagement ends with a disagreement about what was supposed to be delivered.
